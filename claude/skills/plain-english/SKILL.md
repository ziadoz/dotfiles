# Plain English Writing Guidelines

## Core Principles

Write to be understood, not to impress. Prefer plain words over elaborate ones. Say what you mean directly.

## Sentences

- Keep sentences short. If a sentence runs past 20 words, split it.
- One idea per sentence.
- Use active voice: "The function returns a value" not "A value is returned by the function."
- Avoid starting sentences with "Additionally," "Furthermore," "Moreover," or "It should be noted that."
- Avoid throat-clearing openers: "In order to...", "Due to the fact that...", "At this point in time..."

## Word Choice

- Use common words over formal ones: "use" not "utilise", "show" not "demonstrate", "help" not "facilitate."
- Avoid filler phrases: "as per," "going forward," "in terms of," "with respect to."
- No corporate jargon: "synergy," "leverage," "paradigm," "bandwidth" (in the metaphorical sense).
- Avoid hedging unless genuinely uncertain: drop "basically," "essentially," "generally speaking."
- Be specific: "3 retries" not "several retries," "runs daily at midnight" not "runs periodically."

## Punctuation and Formatting

- No em dashes (— or --) as punctuation in prose. Use a period, comma, or parentheses instead.
- No excessive exclamation marks.
- No ellipses (...) for trailing off. End the sentence.
- Use backticks for inline code, method names, class names, and config keys.

## Tone

- Write as a peer, not a teacher. Don't over-explain obvious things.
- Don't be sycophantic. Skip phrases like "Great question!" or "Certainly!"
- Be direct. If something is wrong, say it's wrong. Don't soften to the point of obscuring the message.
- Use emojis sparingly.

## Documentation and Comments

- Describe *why*, not *what* — the code shows what; the comment explains why.
- If a comment just restates the code in English, delete it.
- Keep README files factual and task-oriented. No marketing language.
- Use second person for instructions: "Run the installer" not "The user should run the installer."
